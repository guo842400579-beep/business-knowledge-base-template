## 二、知识库架构

```text
<业务知识库名称>/
├── AGENTS.md              # 本文件，AI 必读
├── README.md              # 给人看的模板使用说明
├── RAW_SOURCES/           # 原始来源，只增不删
│   ├── 设计文档/           # 需求文档、模板、字段说明等
│   ├── 专家知识/           # 业务专家口头/文字补充原文
│   ├── 截图/               # 页面截图、弹窗截图、流程截图
│   └── 用户反馈/           # 用户对 AI 回答或文档的纠正原文
├── WIKI/                  # AI 维护的结构化知识
│   ├── index.md            # 全局总索引
│   ├── log.md              # 变更日志
│   ├── 概念/               # 核心概念和术语
│   ├── 功能模块/           # 系统页面、菜单、按钮、字段
│   ├── 操作流程/           # 跨页面、跨角色的完整流程
│   └── 源码映射/           # 页面动作、接口、源码和规则证据链
├── skills/                # 本知识库本地 Skill
│   ├── query/              # 查询规范
│   ├── lint/               # 健康检查脚本和规则
│   ├── record-feature-to-wiki/ # 系统页面采集并沉淀到 WIKI
│   └── extract-source-to-wiki/ # 源码分析并沉淀源码映射
└── 文档备份/               # 删除或大改前的备份
```

### 目录结构规则

| 场景 | 推荐结构 | 示例 |
|------|----------|------|
| 系统菜单层级清晰 | 跟随侧边栏菜单层级嵌套 | `功能模块/示例系统A/一级菜单/二级菜单/页面.md` |
| 系统菜单较浅或变化频繁 | 一级菜单建文件夹，页面平铺 | `功能模块/示例系统B/一级菜单/页面.md` |
| 页面包含多个 Tab | 父页面建 `_index.md`，每个 Tab 独立文档 | `页面名/页面名_index.md`、`TabA.md` |

> 此处可维护：不同系统的目录落位规则、命名习惯、菜单层级深度、截图归档路径。

### 菜单结构对齐原则

以下三套目录尽量使用同一套系统/菜单/页面层级，便于后续追溯：

| 结构 | 对齐方式 |
|------|----------|
| `RAW_SOURCES/截图/` | `系统/一级菜单/二级菜单/页面/截图文件` |
| `WIKI/功能模块/` | `系统/一级菜单/二级菜单/页面.md` |
| `WIKI/源码映射/` | 镜像 `WIKI/功能模块/`，按页面或动作记录接口、类、方法和版本 |

如果当前知识库没有源码或不需要源码分析，可以保留 `WIKI/源码映射/源码映射_index.md` 作为空入口，并在 index 中标注“未启用”。

## 三、维护入口

知识库维护分为三种模式。无论哪种模式，最终都要保证 RAW 可追溯、WIKI 可查询、index/log/lint 闭环。

| 模式 | 用户给出的信息 | AI 的职责 | 适用 Skill |
|------|----------------|-----------|------------|
| 系统页面实采模式 | 用户给出系统页面、菜单路径或页面名称，要求记录功能 | 打开页面、截图、读取字段和按钮，把证据放入 RAW，再写 WIKI/index/log，运行 lint | `record-feature-to-wiki` + `lint` |
| 源码分析模式 | 用户给出本地源码、接口定义或算法服务代码 | 只读源码，建立页面到代码映射，把业务结论回写到功能模块/概念/流程 | `extract-source-to-wiki` + `lint` |
| RAW 提取模式 | 用户已把操作手册、QA 集、截图、表格、专家补充等放入 `RAW_SOURCES/` | 读取 RAW，提炼结构化知识，更新 WIKI/index/log，运行 lint | `query` + `lint` |

### 信息来源优先级

1. **页面实采优先**：当前系统页面可见的菜单、字段、按钮、弹窗、提示和线上行为是第一事实来源。
2. **源码分析补充**：源码用于解释隐藏校验、接口链路、状态机、算法规则或页面背后逻辑。源码映射只保存证据链，业务结论必须回写到功能模块、概念或流程页。
3. **RAW 提取补充**：操作手册、QA 集、设计文档、表格、历史截图和专家补充用于补全背景、规则和例外。若 RAW 与页面实采冲突，先标记待确认，不直接覆盖页面事实。

### RAW_SOURCES 只增不删

- `RAW_SOURCES/` 中的内容绝不修改、绝不删除，只追加。
- 用户原话、截图、下载模板、页面采集记录都属于原始证据。
- 用户反馈未经总结和确认，不得直接改写成业务结论。
- 对外分享模板时，不要把真实 RAW 一起分享。

## 四、如何检索

### Index 优先原则

任何查询前必须先读：

| 文件 | 作用 |
|------|------|
| `WIKI/index.md` | 全库总索引，建立结构视图 |
| `WIKI/概念/概念_index.md` | 概念入口，建立术语和关系视图 |

如果用户使用口语化称呼，先查 `WIKI/概念/术语对照表.md`，再定位正式文档。

### 检索策略

```text
1. 读 WIKI/index.md + WIKI/概念/概念_index.md
2. 提取问题关键词：系统名、角色、概念、页面名、动作名
3. 定位目录并读取对应 _index.md
4. 读取最相关的 2-5 个文档
5. 信息不足时最多迭代检索 3 轮
6. 回答时注明 WIKI 来源；如查看 RAW，也注明 RAW 来源
```

## 五、如何维护

### 维护前置要求

1. 阅读 `AGENTS.md`。
2. 阅读 `WIKI/index.md` 和 `WIKI/概念/概念_index.md`。
3. 阅读目标目录 `_index.md` 和即将更新的当前 WIKI 文件。
4. 判断本次属于系统页面实采模式、源码分析模式还是 RAW 提取模式。
5. 编辑前确认当前文件真实内容，不凭记忆修改。

### 系统页面实采模式

```text
1. 加载 record-feature-to-wiki skill
2. 打开页面，确认 URL、标题、菜单路径、当前环境/组织信息
3. 采集原始证据：截图、弹窗、侧拉框、下拉、宽表、文件选择器、下载模板
4. 将原始证据归档到 RAW_SOURCES/截图 或 RAW_SOURCES/设计文档
5. 提取结构化知识：筛选条件、按钮功能、列头、Tab、提示、权限/风险点
6. 创建或更新 WIKI 页面
7. 更新相关 index 文件和概念页
8. 追加 WIKI/log.md
9. 运行 lint 验证
```

### 源码分析模式

```text
1. 加载 extract-source-to-wiki skill
2. 确认源码仓库路径、分支、commit 和工作区状态
3. 按页面、按钮、接口或规则收敛分析范围
4. 追踪页面动作到 Controller/Service/DTO/枚举/算法入口
5. 在 WIKI/源码映射/ 中记录证据链
6. 把业务结论回写到功能模块、概念或操作流程
7. 如源码与页面实采或 RAW 冲突，先记录待确认问题，不直接覆盖
8. 更新 index/log
9. 运行 lint 验证
```

### RAW 提取模式

```text
1. 定位并阅读 RAW_SOURCES 中的目标材料
2. 结合现有 WIKI、概念页和 index 判断语境
3. 提取字段、规则、流程、页面结构、角色职责、跨模块关系
4. 创建或更新 WIKI 页面
5. 更新相关 index 文件和概念页
6. 追加 WIKI/log.md
7. 运行 lint 验证
```

RAW 提取适合处理操作手册、QA 集、设计文档、模板、表格、历史截图和专家补充。提取时要保留来源路径，避免把示例数据写成枚举全集。用户纠正先原文记录到 `RAW_SOURCES/用户反馈/`，后续经总结和确认后再写入 WIKI。

### 页面操作权限规则

- “提交、确定、保存、删除确认”等会改变系统数据的按钮，点击前必须向用户申请权限。
- 申请权限时说明当前系统、页面、环境或组织标识；如标识敏感，只在对话中确认，不写入可分享模板。
- “取消、关闭、返回、展开、收起、查看下拉、打开弹窗但不提交”等不落库动作不需要单独申请。
- 即使用户允许操作，也优先停在二次确认层记录风险提示，避免无意义改动。

## 六、文档规范

### Frontmatter

```yaml
---
title: 页面标题
type: concept | module | process | index
source: |
  RAW_SOURCES/截图/示例路径/示例截图.png
  RAW_SOURCES/专家知识/示例补充.md
date: 2026-07-02
updated: 2026-07-02
tags: [示例标签]
related: [相关文档1, 相关文档2]
---
```

- `_index.md` 和 `WIKI/index.md` 可以不写 `source`。
- 普通页面必须写 `source`，确保可追溯。
- 示例数据只能作为示意，不得当成枚举全集。

### 命名规则

| 对象 | 规则 |
|------|------|
| 系统页面 | 必须匹配系统侧边栏或页面上的实际名称 |
| 概念文档 | 使用业务正式称呼；口语称呼写入术语对照表 |
| 流程文档 | 使用“动作 + 流程”命名 |
| 截图目录 | 按系统/菜单/页面层级归档 |
| 源码映射 | 镜像功能模块路径，必要时按页面动作拆分 |

### 数据类型区分

| 内容 | 通用性 | 处理方式 |
|------|--------|----------|
| 列头、字段名、按钮名、状态机、规则条件 | 通用 | 完整记录 |
| 具体行数据、人员姓名、账号、金额、单号 | 示例或敏感 | 脱敏、只留必要示意，标注“示例数据” |

### 流程文档必须包含

- 概述
- 流程图
- 前置条件
- 详细步骤
- 后续追踪
- 常见问题
- 相关页面链接

### 跨模块链接

若两个页面或概念存在依赖关系：

- 两边的 frontmatter `related` 互指。
- 正文中使用交叉引用：`> 关联：参见 [目标页面](路径)`。

### Tab 和按钮文档化

- 含多个 Tab 的页面：入口 `_index.md` 记录 Tab 列表和完成状态，每个 Tab 独立建文档。
- 页面内少量按钮：写入父页面章节。
- 按钮多且逻辑独立：父页面改为 `_index.md`，按钮或 Tab 独立建文档。

## 八、维护铁律

1. 保持 `RAW_SOURCES/` 不可变：只追加，不修改，不删除。
2. 每次维护后更新 `WIKI/index.md` 和相关 `_index.md`。
3. 每次维护后追加 `WIKI/log.md`。
4. 删除或大改文件前，先复制到 `文档备份/`。
5. 编辑前必须读取当前文件。
6. 含表格的 Markdown 优先整文件重写，避免局部 patch 破坏表格。
7. 对外分享前执行敏感词和真实名称检查。

### 日志格式

```markdown
## [YYYY-MM-DD] 操作类型 | 标题
- 操作说明 1
- 操作说明 2
```

操作类型建议固定为：`init` / `ingest` / `query` / `lint` / `update` / `refactor`。

## 九、高频踩坑

| # | 陷阱 | 现象 | 正确做法 |
|---|------|------|----------|
| 1 | 跳过 index | 文档写到错误目录 | 先读 `WIKI/index.md` 和目标 `_index.md` |
| 2 | 把示例数据当全集 | 回答过度泛化 | 标注“示例数据”，只沉淀可复用规则 |
| 3 | 只看首屏 | 漏掉宽表、Tab、按钮 | 横向滚动、展开更多条件、逐个检查 Tab |
| 4 | 只写结论不存 RAW | 无法追溯 | 先归档原始证据，再写 WIKI |
| 5 | 修改真实系统数据 | 造成业务影响 | 落库动作前必须申请权限 |
| 6 | 忘记同步索引 | 查询找不到新文档 | 更新最近 `_index.md`、上层 index、`WIKI/index.md` |
| 7 | 模板夹带业务知识 | 对外泄露 | 使用占位符和虚构示例，分享前敏感词扫描 |
| 8 | 菜单、截图、源码映射目录不一致 | 后续追溯困难 | 首轮建库先记录菜单树，并让三套目录保持同层级 |
| 9 | 源码结论直接覆盖页面事实 | 文档和线上行为冲突 | 页面实采优先，源码冲突先记录待确认 |

```bash

python3 skills/lint/scripts/lint_wiki.py

# 真实建库后检查是否仍有模板占位
python3 skills/lint/scripts/lint_wiki.py --strict-init

# 搜索表格格式错误
rg -n "\|\|\||\|\| \|" WIKI

# 分享前做敏感信息检查，关键词请替换为你的真实敏感词
rg -n "公司名|系统域名|人员名|手机号|邮箱|合同号|订单号" .

# 分享前检查本地杂文件
find . -name ".DS_Store" -o -name ".idea"
```
