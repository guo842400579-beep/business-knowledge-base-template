## 二、知识库架构

```text
<业务知识库名称>/
├── AGENTS.md                    # 本文件，AI 必读
├── README.md                    # 给人的使用说明
├── BOOTSTRAP_ONCE.md            # 仅首次建库使用，完成后删除或归档
├── SHARING_CHECKLIST.md         # 对外分享前的文本、资产和历史检查
├── .work/                       # 临时工作层，Git 忽略
│   ├── page-capture/            # 页面采集队列、候选证据、临时 DOM、恢复点
│   └── source-analysis/         # 源码清单、覆盖矩阵、批次、决策与终审
├── RAW_SOURCES/                 # 永久证据层，只增不删
│   ├── 设计文档/
│   ├── 专家知识/
│   ├── 截图/
│   └── 用户反馈/
├── WIKI/                        # 默认业务查询层
│   ├── index.md
│   ├── log.md                   # 极简审计层，不是业务正文
│   ├── 概念/
│   ├── 功能模块/
│   ├── 操作流程/
│   ├── 源码映射/
│   └── 待确认项/
├── skills/
│   ├── query/
│   ├── lint/
│   ├── record-feature-to-wiki/
│   ├── extract-source-to-wiki/
│   └── orchestrate-source-maintenance/
├── 知识库测试用例/              # 可选的查询回归用例
└── 文档备份/                    # 删除或大改前的备份
```

### 四层信息分离

| 层级 | 内容 | 生命周期 | 默认业务检索 |
|------|------|----------|--------------|
| `.work/` 临时工作层 | 进度、按钮清单、候选截图、临时 DOM、源码清单、恢复点、终审回执 | 中断时保留，发布完成后按批次清理 | 否 |
| `RAW_SOURCES/` 证据层 | 最小充分截图、模板、用户原话、专家原文、必要 DOM | 永久，只增不删 | 否 |
| `WIKI/` 知识层 | 稳定字段、规则、状态、流程、按钮行为和必要信息边界 | 长期维护 | 是 |
| `WIKI/log.md` 审计层 | 变更范围和关键结论，每次最多两条 bullet | 永久 | 否 |

WIKI 必须能脱离维护过程独立阅读。不得把“本轮如何点击”“截图已归档”“未点击保存”“运行了什么工具”等执行流水写入 WIKI。

### 目录对齐原则

以下结构尽量使用同一套系统 / 菜单 / 页面层级：

| 结构 | 对齐方式 |
|------|----------|
| `.work/page-capture/` | 按批次和当前业务对象组织，不作为永久路径 |
| `RAW_SOURCES/截图/` | `系统/一级菜单/二级菜单/页面/证据文件` |
| `WIKI/功能模块/` | `系统/一级菜单/二级菜单/页面.md` |
| `WIKI/源码映射/` | 页面直接镜像系统/菜单路径；概念、流程用同名子目录，记录接口、类、方法、版本和差异 |

系统菜单层级稳定时跟随侧边栏嵌套；菜单较浅或变化频繁时可平铺，但必须在本文件写清各系统的落位规则。

## 三、维护入口

| 模式 | 典型输入 | 主要职责 | Skill |
|------|----------|----------|-------|
| 系统页面实采 | 页面、菜单路径或页面名称 | 安全探索页面，在 `.work` 筛选证据，提升最小 RAW，写 WIKI/index/log | `record-feature-to-wiki` + `lint` |
| 单对象源码分析 | 一个页面、按钮、配置、概念或业务链路 | 固定源码版本，建立映射，把业务结论就近回写知识层 | `extract-source-to-wiki` + `lint` |
| 大型源码维护 | 全仓、多仓、整体覆盖、长期恢复或增量复核 | 建立清单、覆盖矩阵、批次、恢复点和独立终审，再逐对象分析 | `orchestrate-source-maintenance` |
| RAW 提取 | 操作手册、QA、截图、表格、专家补充、用户反馈 | 保留原始来源，提炼稳定知识，更新 WIKI/index/log | 按本文件执行 + `lint` |

### 信息来源与冲突

1. 当前系统页面可见的菜单、字段、按钮、弹窗、提示和线上行为，是页面事实的首要来源。
2. 源码用于解释隐藏校验、状态机、接口链路、算法、配置消费和后台处理；必须绑定仓库、分支和 commit。
3. RAW 用于补充背景、规则、例外和原始口径。
4. 用户明确确认的正式业务规则可以成为业务事实，但不能自动证明某个页面或冻结源码已经按该规则实现。

页面、RAW、用户口径和源码冲突时，必须分开记录：

- 正式业务规则；
- 当前页面行为；
- 当前源码实现；
- 实现差异和验证边界。

不得静默覆盖。需要决策的问题进入 `WIKI/待确认项/`；采集进度和恢复点仍只放 `.work/`。

### RAW 只增不删

- 不修改、不覆盖、不删除 RAW 原文件。
- 页面采集先在 `.work/page-capture/` 保存候选证据，筛选后才提升到 RAW。
- 只提升能独立支撑最终结论的最小集合。
- 不提升导航过程、重复截图、定位控件用 DOM、失败尝试、按钮清单、完成凭证和工具报错。
- 源码分析不复制整仓代码到 RAW；用仓库路径、分支、commit 和关键文件建立证据链。

## 四、如何检索

### 必读入口

任何查询前先读：

| 文件 | 作用 |
|------|------|
| `WIKI/index.md` | 建立全库结构视图 |
| `WIKI/概念/概念_index.md` | 建立术语、概念和关系视图 |

涉及口语、旧称或页面别名时，再读 `WIKI/概念/术语对照表.md`。涉及隐藏逻辑时读源码映射索引；涉及未决口径时读待确认项索引。

### 默认检索边界

业务查询默认只读 `WIKI/index.md`、相关 `_index.md`、概念、功能模块和操作流程正文。

默认排除：

- `.work/`
- `WIKI/log.md`
- `RAW_SOURCES/`
- `文档备份/`
- 源码仓库中的临时扫描结果

仅当用户要求证据或历史、WIKI 有歧义、正文存在信息边界，或需要核对源码版本时，才按 `source` 或映射页定向读取证据。待确认项只能限制答案，不能当作已确认事实。

### 检索步骤

```text
1. 读全局 index 和概念 index
2. 提取系统、角色、概念、页面、动作、字段、状态等关键词
3. 沿相关 _index.md 定位 2–5 个具体文档
4. 最多迭代 3 轮
5. 先给结论，再注明 WIKI 来源、证据边界和仍待确认部分
```

## 五、如何维护

### 通用前置要求

1. 阅读 `AGENTS.md`。
2. 阅读 `WIKI/index.md`、`WIKI/概念/概念_index.md`。
3. 阅读目标目录 `_index.md` 和即将更新的当前文件。
4. 判断维护模式和业务边界。
5. 编辑前确认文件真实内容，不凭记忆修改。
6. 发现用户已有改动时保留并绕开，不擅自回滚或覆盖。
7. 真实建库前先完成 `BOOTSTRAP_ONCE.md`；开始业务维护前 `python3 skills/lint/scripts/lint_wiki.py --strict-init` 必须通过。维护模板本身除外。

### 系统页面实采

```text
1. 在 .work/page-capture/<YYYY-MM-DD>-<范围>/ 建立队列和恢复点
2. 打开一个页面或信息单元，确认标题、URL、菜单路径和当前角色/环境
3. 穷尽安全控件、Tab、宽表、下拉、弹窗、侧拉框和状态
4. 把候选截图、DOM、按钮清单和失败尝试保留在 .work
5. 筛选最小充分证据并提升到 RAW
6. 把稳定事实写入 WIKI，更新相关 index 和极简 log
7. 运行 lint、纯化检查和 git diff --check
8. 完成当前信息单元后再进入下一个
```

容量建议：

- 简单只读列表：同一对话最多 3–4 个页面；
- 有 Tab、弹窗或行操作：最多 2 个页面；
- 复杂联动、状态流转或多步骤表单：每次 1 个页面。

中断时保留 `.work` 恢复点；恢复时先读已写入的 RAW/WIKI 和队列，不凭上下文记忆补写。

#### 页面操作安全边界

- 展开、收起、查看下拉、切换 Tab、打开普通弹窗但不提交、取消、关闭、返回等只读或可逆动作可以直接探索。
- “保存、提交、确定、确认、删除确认、批量生效”等最终写入动作，执行前必须向用户申请明确权限。
- 即使获得权限，也优先停在二次确认层记录风险提示，避免无意义改动。
- “撤回、作废、启停、重试、删除”等状态按钮可能首击即写入。只有确认首击仅打开二次确认层时才可打开；风险不明则停在点击前。
- 文件选择、上传、导入、发送、同步、生成、测试连接、标记已读和外部跳转不能默认视为只读；首击副作用不明时停在点击前。
- 导出、下载和模板下载会在本机落盘，可能带出真实明细；需要时先进入 `.work`，检查并脱敏后再决定是否提升 RAW。
- 不得仅凭按钮名称推断功能；安全入口应实际打开并记录真实 UI 形态。

### 单对象源码分析

1. 把范围收敛到一个页面、按钮、配置、概念或链路。
2. 记录仓库路径、分支、commit 和工作区状态。
3. 按入口 → Controller/API → Service/Rule → DTO/Enum → 持久化/消息/下游消费追踪。
4. 源码映射只放技术证据；业务结论回写原字段、按钮、状态、流程、权限或下游章节。
5. 禁止创建泛化的“源码提取补充”章节堆放所有 finding。
6. 配置类页面必须同时追踪维护入口、保存校验、下游消费者、多场景取数、优先级和兜底。
7. 更新 index/log，运行 lint，并向大型调度任务返回覆盖矩阵回执。

源码能力至少分为：页面已提供、页面未提供、页面未实采、其他业务分支、旧逻辑/死代码、与页面冲突。

### 大型源码维护

全仓、多仓、跨系统或长期任务必须先使用 `orchestrate-source-maintenance`：

- 在 `.work/source-analysis/<scope>/` 冻结范围和源码版本；
- 建立仓库清单、源码入口清单、WIKI 覆盖矩阵、跨对象链路矩阵、未归属入口、批次和决策记录；
- 每个批次按单对象闭环串行发布；
- 每批记录恢复点；
- 源码变化后只把受影响对象标为 `stale`，不要无差别全库重做；
- 分别报告 WIKI 对象、源码入口和跨对象链路覆盖率。

库存脚本只生成候选导航，不能替代技术栈人工盘点。文件名启发式之外还要核对路由/Resolver、proto/schema、消息与任务配置、存储过程、构建清单和运行装配。脏源码工作树必须记录完整 commit 与工作区指纹；无法复核时不得标为 `published`。

`published` 只表示该对象的源码映射、业务回写、index、log、lint 和覆盖矩阵回执都已闭环。`blocked-conflict` 不是完成状态，除非明确批准从本轮范围排除并记录理由。

大型任务结束必须生成独立 `final-audit.md`，记录基线 commit 或 WIKI tree hash。基线之后有业务文档变化时，该终审自动视为 `stale`，不得继续当作当前工作树证明。

### RAW 提取

1. 定位并只读目标 RAW。
2. 结合现有 WIKI、概念页和 index 判断语境。
3. 提取字段、规则、流程、角色、状态和跨模块关系。
4. 用户专业补充保留原文；用户纠正先归档原话。
5. 只把确认后的稳定结论写入 WIKI。
6. 更新相关 index/log 并运行 lint。

PDF、Excel 等文件必须使用对应文件处理规范，不能把示例数据推断为枚举全集。

### 待确认项

`WIKI/待确认项/` 只记录影响业务结论的人工决策。主状态使用：

- `待业务确认`
- `待页面实采`
- `待运行验证`
- `已确认待回写`
- `已关闭`

进度、源码覆盖和人工决策是三套状态，禁止混用。状态和字段规范见 `WIKI/待确认项/待确认项_index.md`。

## 六、文档规范

### Frontmatter

```yaml
---
title: 页面标题
type: concept | module | process | source-map | decision | index
source: |
  RAW_SOURCES/截图/示例路径/示例截图.png
date: 2026-07-23
updated: 2026-07-23
tags: [示例标签]
related: [相关文档1, 相关文档2]
---
```

- `WIKI/index.md`、`_index.md` 和其他 `type: index` 页面可以不写 `source`。
- 普通页面必须写非空 `source`，且每项必须指向 `RAW_SOURCES/` 下存在的具体文件。
- `source` 禁止绝对路径、目录路径、路径越界和伪造来源。
- `type` 只能是 `concept`、`module`、`process`、`source-map`、`decision`、`index`。
- 源码映射正文使用 `source-map`；待确认单项使用 `decision`。两者都必须引用具体 RAW，不能伪装成 `index` 绕过证据校验。

### 命名与结构

| 对象 | 规则 |
|------|------|
| 系统页面 | 匹配当前系统实际名称 |
| 概念文档 | 使用正式称呼；口语和旧称写入术语对照表 |
| 流程文档 | 使用“动作 + 流程”命名 |
| 截图目录 | 按系统/菜单/页面层级归档 |
| 源码映射 | 页面直接镜像系统/菜单；概念和流程分别放入 `概念/`、`操作流程/` |

### Index

- `_index.md` 只记录稳定结构、入口和功能摘要。
- 不写采集进度、实采日期、完成打勾、恢复点或临时受阻原因。
- 同目录普通文档和直接子目录入口必须能从 index 通过链接到达。
- 新增、移动或重命名后同步更新最近 index、上层 index 和 `WIKI/index.md`。

### Tab 与按钮

- 多 Tab 页面：父 `_index.md` 记录稳定 Tab 列表和摘要，每个复杂 Tab 独立成文；完成状态放 `.work`。
- 普通弹窗、确认框、单步骤操作写入父页面。
- 完整新页面、复杂侧拉框、多步骤流程、独立审核或状态流转可独立成文。
- 上级 index 只展示真实菜单，不把页面内按钮伪装成同级菜单。

### 流程文档

`type: process` 页面必须包含：

- 概述
- 流程图
- 前置条件
- 详细步骤
- 后续追踪
- 常见问题
- 相关页面链接

### 数据与敏感信息

| 内容 | 处理方式 |
|------|----------|
| 字段名、列头、按钮名、状态机、规则条件 | 完整记录 |
| 具体人员、账号、客户、金额、单号、手机号、邮箱 | 删除、脱敏或仅留必要示意 |
| 当前列表值、下拉样本、测试数据 | 标注“示例数据”，不当作全集 |

永久证据提升前必须逐图、逐文件检查敏感信息，在 `.work` 生成脱敏副本；使用唯一文件名，目标已存在时拒绝覆盖，并记录候选来源、RAW 目标和 SHA-256。

### 日志

```markdown
## [YYYY-MM-DD] 操作类型 | 标题
- 变更范围。
- 关键业务结论或遗留边界。
```

操作类型固定为：`init` / `ingest` / `query` / `lint` / `update` / `refactor`。每次最多两条 bullet，不复制字段、按钮、截图路径、工具过程或完成凭证。

## 七、发布与验证

### 确定性发布门

```bash
python3 skills/lint/scripts/lint_wiki.py
```

默认 lint 检查 frontmatter、source、断链、index、表格、行号污染和明确过程噪声。出现 P0/P1/P2 时不得称为“发布检查通过”。

### 非阻断内容审计

```bash
python3 skills/lint/scripts/lint_wiki.py --audit
```

`--audit` 汇总成熟度、可能的过程语言、正文偏短、索引时效和状态冲突。Audit 提示需要人工判断，不阻断发布，也不能自动等同于知识缺失。

### 页面纯化

```bash
python3 skills/record-feature-to-wiki/scripts/check_wiki_purity.py WIKI/本次修改文件.md
```

硬性过程噪声会失败；“当前样本”“待复核”等可能是合法信息边界，只提示人工判断。

### 初始化与分享

```bash
python3 skills/lint/scripts/lint_wiki.py --strict-init
python3 skills/lint/scripts/lint_wiki.py --share-check
python3 skills/lint/scripts/lint_wiki.py \
  --sensitive-term "公司名" \
  --sensitive-term "系统域名"
```

- `--strict-init` 用于从模板创建真实知识库后检查占位残留；原始模板运行时预期会提示。
- `--share-check` 检查 `.DS_Store`、`.idea` 等本地杂文件。
- 敏感词扫描不能替代图片、PDF、Office、备份和 Git 历史的人工复核。

修改后还要运行：

```bash
git diff --check
```

## 八、维护铁律

1. 先读 AGENTS 和 Index，再查询或修改。
2. RAW 只增不删；候选证据先在 `.work` 筛选。
3. WIKI 只写稳定知识，不写维护流水。
4. 页面事实、正式业务规则和源码实现分层表达。
5. 源码映射保存证据，业务结论就近回写知识层。
6. Index 不保存临时进度。
7. Log 每次最多两条。
8. 编辑前读取当前文件。
9. 含表格的 Markdown 优先整文件重写。
10. 删除或大改前先备份到 `文档备份/`。
11. 写入系统数据前申请权限；风险不明的首击写入按钮不得探索。
12. 大型任务原地续跑，恢复点和覆盖矩阵只留 `.work`。
13. `blocked-conflict` 不等于完成。
14. 旧终审在基线变化后自动过期。
15. 对外分享前执行文本、文件名、图片、二进制资产、备份和历史的多层脱敏检查。

## 九、高频踩坑

| # | 陷阱 | 正确做法 |
|---|------|----------|
| 1 | 跳过 index，文档落错目录 | 先读全局和目标 `_index.md` |
| 2 | 候选证据全量进 RAW | 先在 `.work` 筛选最小充分集合 |
| 3 | WIKI 混入按钮清单和点击过程 | 过程留 `.work`，WIKI 改写成稳定事实 |
| 4 | Index 记录完成打勾或实采日期 | Index 只写稳定结构和摘要 |
| 5 | 把代码存在写成页面可用 | 分类页面已提供、未提供、未实采和旧逻辑 |
| 6 | 源码结论只留映射页 | 就近回写原字段、按钮、状态、流程或概念 |
| 7 | 配置页只追保存接口 | 继续追下游消费者、优先级和兜底 |
| 8 | 把待确认项当业务事实 | 只用来限定答案，确认后再回写 |
| 9 | 把 Audit 提示当发布失败 | 默认 lint 与非阻断 audit 分开报告 |
| 10 | 用扫描文件数宣称全量完成 | 分别核算 WIKI 对象、源码入口和跨对象链路 |
| 11 | 把旧终审当永久证书 | 校验基线 commit/tree hash，变化后标 stale |
| 12 | 分享时只扫关键词 | 同时复核图片、二进制文件、备份、`.work` 和 Git 历史 |
