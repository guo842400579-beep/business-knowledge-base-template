# Record Feature To Wiki

## 核心原则

把“当前系统页面可见事实”变成“可追溯、可查询、可维护”的 WIKI。页面实采优先于历史文档和源码推断；源码只用于解释隐藏逻辑，不得覆盖页面事实。

本 skill 是模板化流程，不绑定任何真实系统。使用时把“目标系统、目标页面、目标角色、目标组织”替换为当前业务知识库的真实上下文；对外分享模板前不得保留真实人员、域名、账号、组织、单号、金额等敏感信息。

## 前置硬规则

开始任何维护前必须读取：

| 文件 | 目的 |
|------|------|
| `AGENTS.md` | 确认业务边界、证据优先级、目录规则和权限规则 |
| `WIKI/index.md` | 确认全库结构 |
| `WIKI/概念/概念_index.md` | 确认术语和概念上下文 |
| 目标目录 `_index.md` | 确认页面落位和已有文档 |
| 即将更新的 WIKI 文件 | 避免凭记忆覆盖用户已有结论 |

如果目标 WIKI 文件不存在，先读最近上级 `_index.md`，再决定新建路径。

## 强制执行协议：单页面闭环

记录功能信息时，每次只处理一个页面、Tab、弹窗、侧拉框、配置项或按钮组。

当前信息单元必须完成以下闭环后，才能进入下一个：

1. 打开并确认页面或控件。
2. 采集字段、按钮、提示、状态、截图和必要下载物。
3. 归档 RAW 证据。
4. 创建或更新对应 WIKI。
5. 更新相关 index 和 `WIKI/log.md`。
6. 运行 lint 或说明未运行原因。

允许例外：只做首次菜单树梳理时，可以快速打开多个页面建立目录骨架；一旦开始记录字段、按钮、规则、弹窗、宽表或流程，必须切回单页面闭环。

## 页面采集范围

### 1. 页面确认

记录：

| 信息 | 写法 |
|------|------|
| 页面标题 | 按页面可见文字记录 |
| 菜单路径 | 按系统真实侧边栏或导航层级记录 |
| URL | WIKI 中写脱敏版本；敏感参数只在对话确认，不写入模板 |
| 环境/组织 | 如敏感，只用于权限确认，不写入可分享文档 |
| 角色/权限 | 记录角色类别，不记录真实姓名、账号或敏感权限细节 |

### 2. 原始证据归档

按 `AGENTS.md` 的目录结构归档：

| 证据 | 推荐路径 |
|------|----------|
| 页面截图、弹窗、侧拉框、下拉、宽表 | `RAW_SOURCES/截图/<系统>/<菜单>/<页面>/` |
| 下载模板、字段说明、导入样例 | `RAW_SOURCES/设计文档/<系统>/<页面>/` |
| 专家口头或文字补充 | `RAW_SOURCES/专家知识/` |
| 用户纠正或反馈原文 | `RAW_SOURCES/用户反馈/` |

RAW 只增不删，不改写原始文件。截图文件名建议包含日期、页面、状态或控件名，例如 `YYYY-MM-DD_页面名_弹窗名.png`。

### 3. 页面结构穷尽

必须检查：

- Tab、子页面、锚点、步骤条。
- 筛选区、更多条件、默认值、placeholder。
- 顶部按钮、批量按钮、单行操作、更多操作下拉。
- 列表列头、固定列、横向滚动后的所有列。
- 分页、排序、空状态、有数据状态、异常状态。
- 弹窗、侧拉框、二次确认、toast、错误提示。
- 下拉联动、日期选择、文件选择器、导入/导出、下载模板。
- 页面内说明文字、风险提示和权限提示。

首屏截图不能代表完整页面；宽表必须横向滚动确认完整列头。

## 操作权限规则

- “提交、确定、保存、删除确认、批量生效”等会改变系统数据的动作，点击前必须向用户申请明确权限。
- 申请权限时说明当前系统、页面、环境或组织标识；敏感标识只在对话中确认，不写入 WIKI。
- “取消、关闭、返回、展开、收起、查看下拉、打开弹窗但不提交”等不落库动作通常可直接执行。
- 删除、提交、确认类动作优先停在二次确认层，记录二次确认文案，不做无意义改动。
- 批量按钮要分别记录未勾选时的提示和勾选后的真实弹窗或风险提示。

## 写入 WIKI

### 1. Frontmatter

普通页面必须包含：

```yaml
---
title: 页面标题
type: module
source: |
  RAW_SOURCES/截图/示例系统/示例菜单/示例页面/YYYY-MM-DD_首屏.png
date: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [示例系统, 示例页面]
related: [相关概念或页面]
---
```

`_index.md` 和 `WIKI/index.md` 可不写 `source`。普通页面如果暂时没有 RAW，不能伪造来源；应先归档证据或明确标注待补证据。

### 2. 页面文档建议结构

```markdown
# 页面标题

## 页面定位

## 入口与菜单路径

## 页面结构

## 筛选条件

## 列表字段

## 按钮与操作

## 弹窗 / 侧拉框 / 下拉

## 状态与提示

## 权限与风险点

## 与其他页面或概念的关系

## 待确认问题
```

按实际页面裁剪，不要为了套模板写空章节。

### 3. 内容写法

- 字段、按钮、状态、规则条件按系统原文记录。
- 示例行数据、人员姓名、账号、金额、单号要脱敏，只保留必要示意并标注“示例数据”。
- 页面有多个 Tab 时，父页面用 `_index.md` 记录 Tab 列表和完成状态，每个 Tab 独立成文。
- 页面按钮少量时写入父页面；按钮多且逻辑独立时，按钮或弹窗独立成文。
- 跨模块有关联时，frontmatter `related` 和正文链接都要互相指向。
- 含表格的 Markdown 优先整文件重写，避免局部 patch 破坏列数。

## 冲突处理

如果页面实采、现有 WIKI、RAW、源码映射或用户反馈冲突：

1. 不直接覆盖。
2. 在当前页面或源码映射页记录“待确认问题”。
3. 向用户列出冲突来源、影响范围和建议处理方式。
4. 用户确认后再改写业务结论。

冲突清单格式：

```markdown
| 编号 | 主题 | 页面实采 | 现有 WIKI/RAW/源码 | 影响范围 | 建议处理 |
|------|------|----------|--------------------|----------|----------|
| C1 | ... | ... | ... | ... | 标为待确认 / 以页面为准 / 需补采 |
```

## 同步索引和日志

每次维护后必须：

- 更新最近目录 `_index.md`。
- 更新上层模块 index。
- 更新 `WIKI/index.md`。
- 追加 `WIKI/log.md`。

日志格式：

```markdown
## [YYYY-MM-DD] ingest|update | 标题
- 新增/更新：WIKI/功能模块/...
- RAW 证据：RAW_SOURCES/...
- 关键变化：...
- 待确认：...
```

## 验证

运行：

```bash
python3 skills/lint/scripts/lint_wiki.py
```

真实建库后或对外分享前运行：

```bash
python3 skills/lint/scripts/lint_wiki.py --strict-init
```

如 lint 发现既有问题，说明哪些与本次变更有关，哪些是历史遗留。
