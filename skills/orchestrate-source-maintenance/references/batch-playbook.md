# 批次、恢复与终审手册

## 目录

1. 批次卡片
2. 恢复点
3. 单批结束条件
4. 扩大范围
5. 独立终审

## 1. 批次卡片

在 `batches.md` 维护：

```markdown
## B001 | 示例主链路

- 状态：planned
- 覆盖行：KB-0001, KB-0002
- 源码版本：repository-a@<commit>
- 工作区：clean；或 dirty@<fingerprint>
- 入口：<Controller/API/Handler>
- 边界：从页面提交到结果落库；不展开外部支付实现
- 页面证据：已实采 / 待实采
- 依赖：示例概念、上游配置
- 恢复点：尚未读取入口
- 发布结果：
```

## 2. 恢复点

只记录：

- 下一步；
- 固定 commit；
- 当前对象和边界；
- 已验证入口；
- 当前阻塞和所需证据。

不复制长代码、完整调用链或业务正文。

## 3. 单批结束条件

- 每个对象绑定源码版本。
- 业务规则已回写最近的业务章节。
- 页面能力分类完整。
- 冲突和待确认项已登记。
- Index、两条以内 log、默认 lint、纯化和 diff 通过。
- 覆盖矩阵、finding、未归属入口和恢复点已更新。
- 相关 `cross-object-chains.csv` 行已更新。

## 4. 扩大范围

只有以下情况才扩大一层：

- 当前链路调用另一仓库并影响状态或结果；
- 公共枚举、规则、配置或远程接口改变当前结论；
- 消息、任务或补偿逻辑决定最终状态；
- 当前页面只是维护入口，真正语义来自多个下游消费者。

扩大后拆出依赖对象或后续批次，不在当前对象无限追踪。

## 5. 独立终审

`final-audit.md` 至少记录：

```markdown
# 最终审计

- 审计日期：
- 知识库 commit：
- 知识库工作区状态：
- wiki_snapshot_sha256：
- 源码仓库与 commit：
- 审计范围：
- 结论：PASS / FAIL / STALE
```

检查：

1. 纳入对象全部终态。
2. 高价值入口无未解释 `unassigned`。
3. Finding 全部使用受控融合动作。
4. 不存在 `append-source-section`、重复权威位置或技术路径泄漏。
5. 源码映射与业务回写一致。
6. 冲突已关闭或批准排除。
7. 默认 lint、纯化和 `git diff --check` 通过。
8. 当前 WIKI 快照与审计基线一致。

终审应使用新的审阅回合重新读取证据并重算快照；优先由未执行批次的代理或人员进行。审计后 WIKI 内容、源码版本、工作区指纹或关键业务证据变化时，把结论改为 `STALE`，只重开受影响对象。
