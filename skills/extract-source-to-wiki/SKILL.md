# 源码分析到知识库

## 核心原则

源码映射是证据层，不是业务知识的最终阅读层。页面可见事实以页面实采为优先；源码用于解释隐藏校验、接口链路、状态机、算法规则和页面背后逻辑。

## 前置要求

开始前必须先读：

| 文件 | 目的 |
|------|------|
| `AGENTS.md` | 确认业务边界、三种信息添加模式和菜单结构对齐规则 |
| `WIKI/index.md` | 建立全库结构视图 |
| `WIKI/概念/概念_index.md` | 建立术语和概念上下文 |
| `WIKI/功能模块/功能模块_index.md` | 确认页面所在菜单结构 |
| `WIKI/源码映射/源码映射_index.md` | 确认源码映射目录和已有证据 |

## 必走流程

1. 收敛范围：一次只分析一个页面、按钮、接口、规则或流程。
2. 记录版本：对每个源码仓库记录路径、分支、commit 和工作区状态。
3. 追踪链路：从页面动作、接口、方法名、字段名或枚举名开始，用搜索定位 Controller/Service/DTO/枚举/算法入口。
4. 写源码映射：在 `WIKI/源码映射/` 下按功能模块路径建文档，记录接口、类、方法、版本和关键逻辑。
5. 回写业务结论：把可读业务结论写回 `WIKI/功能模块/`、`WIKI/概念/` 或 `WIKI/操作流程/`。
6. 处理冲突：如果源码与页面实采、操作手册、QA 集或用户反馈冲突，先写入“待确认问题”，不要直接覆盖正文。
7. 更新索引和日志：同步 `WIKI/源码映射/源码映射_index.md`、相关 `_index.md`、`WIKI/index.md` 和 `WIKI/log.md`。
8. 运行 lint。

## 常见错误

- 把源码映射页写成逐行代码注释。
- 业务结论只留在源码映射页，没有回写到页面、概念或流程。
- 没有记录 commit，导致后续无法判断结论是否仍适用。
- 源码结论和页面实采冲突时静默覆盖页面事实。
