# 源码分析到知识库

## 核心原则

源码映射是证据层，不是业务知识的最终阅读层。

- `WIKI/源码映射/` 记录页面动作、接口、类、方法、版本和证据链。
- 从源码提炼出的可读业务结论，必须回写到 `WIKI/功能模块/`、`WIKI/概念/` 或 `WIKI/操作流程/`。
- 页面可见事实以页面实采为优先；源码用于解释隐藏校验、接口链路、状态机、算法规则和后台处理。
- 代码存在不等于页面可用。必须区分“页面可见”“页面未观察到”“仅代码存在/可能遗留”。
- 不要把源码映射页写成逐行代码注释库。

本 skill 是去业务化模板。不要写入参考项目的真实仓库名、内部域名、人员、任务号或页面专名；使用时替换为当前业务的本地源码路径和通用模块名。

## 何时使用

用户出现以下意图时使用：

- “根据源码补充知识库”
- “这个按钮提交后调了什么接口”
- “隐藏校验/状态机/枚举值/算法规则是什么”
- “页面和代码怎么对应”
- “源码里有哪些 WIKI 没维护的业务逻辑”
- “检查当前 WIKI 是否把代码能力误写成页面功能”

如果用户只是问已有业务知识，先用 `query`。只有 WIKI 不足以回答隐藏逻辑、接口链路或代码原因时，再进入源码分析。

## 前置要求

开始前必须先读：

| 文件 | 目的 |
|------|------|
| `AGENTS.md` | 确认业务边界、信息来源优先级和源码映射规则 |
| `WIKI/index.md` | 建立全库结构视图 |
| `WIKI/概念/概念_index.md` | 建立术语和概念上下文 |
| `WIKI/功能模块/功能模块_index.md` | 确认页面所在菜单结构 |
| `WIKI/源码映射/源码映射_index.md` | 确认源码映射目录和已有证据 |
| 目标目录 `_index.md` 和目标 WIKI 当前文件 | 避免覆盖已有页面事实和用户纠正 |

源码仓库只读。不要在源码仓库内修改代码，除非用户明确要求做代码变更。

## 工作流

### 1. 收敛范围

一次只分析一个对象：

- 一个页面或 Tab。
- 一个按钮或弹窗。
- 一个接口或任务链路。
- 一个字段、枚举、状态机或校验规则。
- 一个算法入口、约束、目标函数或结果回写链路。

范围不清时，先读相关 index 和现有页面文档，列出 2-5 个候选入口，再追踪。

### 2. 记录源码版本

对每个源码仓库记录：

```bash
git rev-parse --short HEAD
git branch --show-current
git status --short
```

写入映射页时至少包含：

| 信息 | 说明 |
|------|------|
| 仓库路径 | 可脱敏为本地相对说明或 `<源码仓库路径>` |
| 分支 | 当前读取分支 |
| commit | 当前读取版本 |
| 工作区状态 | 是否有未提交改动 |
| 关键文件 | Controller、Service、DTO、枚举、算法入口等路径 |

如果工作区不干净，标注“基于当前本地工作区读取”。

### 3. 追踪链路

优先顺序：

1. 页面请求接口、URL、RPC 方法、按钮动作名或字段名。
2. Controller / API / Facade / Application Service。
3. Request / Response / DTO / VO / 枚举。
4. Domain Service / Validator / Rule Handler / 状态机。
5. 算法或任务入口、上下文构建对象。
6. 约束、目标函数、计算规则、求解或处理结果。
7. 持久化、状态查询、结果展示或回写链路。

使用 `rg` 搜索接口、方法名、枚举名、字段名，再局部读取相关文件。不要无目标整仓库复制源码。

### 4. 分类证据

每个发现都要分类：

| 分类 | 写法 |
|------|------|
| 页面可见 | 页面实采或 WIKI 已有入口可对应，源码用于解释规则 |
| 页面未观察到 | 代码存在，但当前页面实采或 RAW 未发现入口 |
| 仅代码存在/可能遗留 | 代码有能力，但无页面、手册或用户证据支持 |
| 与页面冲突 | 源码与页面实采、WIKI、RAW 或用户反馈不一致 |

只有“页面可见”或不推翻现有事实的补充规则，才可直接回写业务文档。其他分类应进入源码映射的差异或待确认记录。

## 源码映射页结构

`WIKI/源码映射/` 应镜像 `WIKI/功能模块/` 的系统/菜单/页面层级。页面或动作级内容拆成独立 Markdown，避免单文件过大。

建议结构：

```markdown
---
title: 页面名源码映射
type: index
date: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [源码映射, 示例系统]
related: [页面名]
---

# 页面名源码映射

## 源码版本

| 仓库 | 分支 | commit | 工作区状态 | 作用 |
|------|------|--------|------------|------|

## 页面动作到源码入口

| 页面动作 | 页面接口 | 后端入口 | 关键逻辑 | 证据分类 | 业务结论去向 |
|----------|----------|----------|----------|----------|--------------|

## 字段 / 枚举 / 状态映射

| 页面字段或状态 | 源码对象 | 处理规则 | 证据分类 | 业务结论去向 |
|----------------|----------|----------|----------|--------------|

## 已回写业务结论

- ...

## 页面未提供或代码-only 能力

- ...

## 待确认问题

- ...
```

如不是页面，可把“页面动作”替换为“功能动作”“接口链路”或“规则入口”，但必须保留源码版本、证据分类、业务结论去向和待确认问题。

## 回写业务文档

根据结论类型写入：

| 结论类型 | 写入位置 |
|----------|----------|
| 页面字段、按钮、弹窗、导入、保存、删除、批量操作规则 | `WIKI/功能模块/...` |
| 跨页面概念、校验口径、枚举含义、算法原则、状态机 | `WIKI/概念/...` |
| 操作顺序、排障路径、提交前后流程 | `WIKI/操作流程/...` |
| 接口、类、方法、代码版本、追踪路径 | `WIKI/源码映射/...` |

回写时优先“就近整合”：

- 字段规则补进字段说明。
- 按钮规则补进按钮或弹窗说明。
- 页面执行规则补进“后台处理规则”“执行口径”等业务小节。
- 跨页面规则补进概念或流程页，并在页面文档中保留简短入口链接。

不要在页面末尾堆一个泛化的“源码提取补充”章节，把所有代码发现塞进去。代码路径细节留在源码映射页。

## 冲突暂停确认

如果源码与现有知识库、页面实采、RAW 或用户反馈冲突，必须暂停业务文档回写。

冲突包括：

| 冲突类型 | 示例 |
|----------|------|
| 源码与 WIKI 结论相反 | WIKI 写全量覆盖，代码显示按条件更新 |
| 源码与页面实采不同 | 页面提示必填，代码允许为空 |
| 代码存在但页面未暴露 | 后端有接口，页面和手册都没有入口 |
| 源码与用户反馈不同 | 用户确认必填，代码当前未校验 |
| 多仓库口径不同 | 上游保存字段，下游处理忽略或重算 |
| 本地源码疑似非线上版本 | 页面行为与本地代码不一致 |

冲突清单格式：

```markdown
发现以下冲突，先不直接改业务文档：

| 编号 | 主题 | 现有知识库/页面说法 | 源码证据 | 影响范围 | 建议处理 |
|------|------|----------------------|----------|----------|----------|
| C1 | ... | ... | 仓库@commit: 文件/类/方法 | ... | 保留页面口径 / 改为源码口径 / 标为待确认 |
```

处理要求：

1. 先写入源码映射页的“待确认问题”或“冲突记录”。
2. 向用户说明冲突来源、影响范围和建议处理方式。
3. 等用户确认后，再更新功能模块、概念或流程文档。
4. 用户暂不确认时，业务文档保留原说法，源码映射保留冲突记录。

## 索引、日志和验证

完成后必须更新：

- `WIKI/源码映射/` 相关各级 `_index.md`。
- 被回写的 `WIKI/功能模块/`、`WIKI/概念/` 或 `WIKI/操作流程/` 相关 index。
- `WIKI/index.md`。
- `WIKI/log.md`。

日志应包含：

- 本次读取的源码仓库、分支、commit。
- 新增或更新的源码映射页。
- 已回写的业务文档。
- 页面未观察到或代码-only 的发现。
- 仍需用户确认的问题。

运行：

```bash
python3 skills/lint/scripts/lint_wiki.py
```

## 常见错误

- 把源码映射页写成逐行代码注释。
- 业务规则只留在源码映射页，没有回写到业务文档。
- 把代码存在的接口写成当前页面按钮或功能。
- 不记录 commit，导致后续无法判断结论是否仍适用。
- 源码结论和页面实采冲突时静默覆盖页面事实。
- 把所有结论堆到页面末尾，而不是嵌入字段、按钮、流程或概念。
- 新增映射页后忘记同步 index 和 `WIKI/log.md`。
