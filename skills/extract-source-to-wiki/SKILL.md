# 源码分析到知识库

## 核心原则

源码映射是证据层，不是业务知识的最终阅读层。

- `WIKI/源码映射/` 记录页面动作、接口、类、方法、版本和证据链。
- 从源码提炼出的可读业务结论，必须回写到 `WIKI/功能模块/`、`WIKI/概念/` 或 `WIKI/操作流程/`。
- 页面可见事实以页面实采为优先；源码用于解释隐藏校验、接口链路、状态机、算法规则和后台处理。
- 代码存在不等于页面可用。必须区分“页面可见”“页面未观察到”“仅代码存在/可能遗留”。
- 不要把源码映射页写成逐行代码注释库。

本 skill 是去业务化模板。不要写入参考项目的真实仓库名、内部域名、人员、任务号或页面专名；使用时替换为当前业务的本地源码路径和通用模块名。

## 任务路由

用户出现以下意图时使用：

- “根据源码补充知识库”
- “这个按钮提交后调了什么接口”
- “隐藏校验/状态机/枚举值/算法规则是什么”
- “页面和代码怎么对应”
- “源码里有哪些 WIKI 没维护的业务逻辑”
- “检查当前 WIKI 是否把代码能力误写成页面功能”

如果用户只是问已有业务知识，先用 `query`。只有 WIKI 不足以回答隐藏逻辑、接口链路或代码原因时，再进入源码分析。

先判断授权模式：

| 模式 | 用户意图 | 允许动作 |
|------|----------|----------|
| 只读诊断 | “解释、分析、查代码、这个按钮调什么接口” | 只读源码并在对话中报告；不修改 WIKI、RAW、Index 或 log |
| 知识库维护 | “补充、维护、更新、沉淀到知识库” | 执行源码映射、业务回写、Index、log 和验证闭环 |

不得把只读诊断自动升级为知识库写入。只读诊断也要报告仓库、完整 commit、工作区状态、证据路径和结论边界。

以下任务不要直接在本 Skill 中铺开：

- 全仓库或多仓库分析；
- 评估整体源码覆盖率；
- 恢复中断的大型源码维护；
- 源码版本变化后的增量复核；
- 需要独立最终审计的长期任务。

这些任务先加载 `../orchestrate-source-maintenance/SKILL.md`，由它冻结范围、源码版本、覆盖矩阵和批次，再把一个覆盖行或一条链路交给本 Skill。

## 前置要求

开始前必须先读：

| 文件 | 目的 |
|------|------|
| `AGENTS.md` | 确认业务边界、信息来源优先级和源码映射规则 |
| `WIKI/index.md` | 建立全库结构视图 |
| `WIKI/概念/概念_index.md` | 建立术语和概念上下文 |
| `WIKI/功能模块/功能模块_index.md` | 确认页面所在菜单结构 |
| `WIKI/源码映射/源码映射_index.md` | 确认源码映射目录和已有证据 |
| `WIKI/待确认项/待确认项_index.md` | 确认人工决策状态和登记规范 |
| 目标目录 `_index.md` 和目标 WIKI 当前文件 | 避免覆盖已有页面事实和用户纠正 |

知识库维护模式还必须先运行：

```bash
python3 skills/lint/scripts/lint_wiki.py --strict-init
```

若 `BOOTSTRAP_ONCE.md` 仍存在、AGENTS 仍有业务占位或检查未通过，只执行初始化或只读诊断，不写业务知识。维护模板本身时不适用此门。

源码仓库只读。除非用户另行明确要求，不运行可能写入构建目录、缓存、锁文件或生成文件的构建、测试、代码生成、格式化、依赖安装和清理命令。

如果任务来自大型源码维护，还要读取覆盖矩阵行、固定 commit、批次边界、已知依赖和恢复点，不自行扩大到无关业务域。

## 工作流

### 1. 收敛范围

一次只分析一个对象：

- 一个页面或 Tab。
- 一个按钮或弹窗。
- 一个接口或任务链路。
- 一个字段、枚举、状态机或校验规则。
- 一个算法入口、约束、目标函数或结果回写链路。

范围不清时，先读相关 index 和现有页面文档，列出 2-5 个候选入口，再追踪。

大型任务分配对象时，必须遵守其覆盖行和批次边界。只有当前结论依赖跨仓状态、公共规则、消息或下游消费者时才扩大一层，并把新增对象拆成依赖行或后续批次。

如果对象是配置类、规则类或基础数据类页面，不要只分析新增/编辑/删除接口。必须同时追踪：

- 维护入口：列表、详情、新增、编辑、复制、启停、导入、导出、删除。
- 保存校验：必填、重复、范围、枚举、批量拆分、逻辑删除、整体替换。
- 下游使用点：该配置在其他页面、流程、任务、算法、结果展示或外部同步中如何被读取和匹配。
- 多消费场景：同一配置被多个流程使用时，必须分别说明各流程的取数口径，不能写成一个统一规则。

### 2. 记录源码版本

对每个源码仓库记录：

```bash
git rev-parse HEAD
git branch --show-current
git status --short
```

写入映射页时至少包含：

| 信息 | 说明 |
|------|------|
| 仓库路径 | 可脱敏为本地相对说明或 `<源码仓库路径>` |
| 分支 | 当前读取分支 |
| commit | 当前读取版本 |
| 工作区状态 | 是否有未提交改动 |
| 工作区指纹 | 脏工作树中 tracked diff 与 untracked 内容的 SHA-256 |
| 关键文件 | Controller、Service、DTO、枚举、算法入口等路径 |

优先分析干净、可复现的 commit。工作区不干净时：

- 不清理或覆盖用户改动；
- 记录完整 `git status --short` 和工作区指纹；
- 在同一工作树上终审时重新计算并核对指纹；
- 无法冻结或复核实际内容时返回 `needs-reproducible-source`，不得标 `published`。

### 3. 追踪链路

页面/接口型对象的优先顺序：

1. 页面请求接口、URL、RPC 方法、按钮动作名或字段名。
2. Controller / API / Facade / Application Service。
3. Request / Response / DTO / VO / 枚举。
4. Domain Service / Validator / Rule Handler / 状态机。
5. 算法或任务入口、上下文构建对象。
6. 约束、目标函数、计算规则、求解或处理结果。
7. 持久化、状态查询、结果展示或回写链路。

消息消费者、定时任务、存储过程、前端状态机、GraphQL resolver 或算法服务不强套 Controller → Service。改用：

```text
触发源 / 调度 / 订阅 / Resolver
→ 入口条件与运行配置
→ 核心规则或计算
→ 持久化 / 消息 / 外部调用
→ 可观察结果、补偿与失败处理
```

使用 `rg` 搜索接口、方法名、枚举名、字段名，再局部读取相关文件。不要无目标整仓库复制源码。

配置类、规则类或基础数据类页面建议采用更收敛的检索路径：

1. 页面路由、Controller、API 或配置入口。
2. Facade / Application Service / Use Case。
3. Domain Service / ServiceImpl / Rule Handler。
4. Mapper / Repository / 自定义 SQL。
5. DTO / VO / PO / Enum / Constant。
6. 下游调用点，例如流程编排、任务处理、结果生成、算法上下文、外部同步。
7. 导入导出工具、进度条、失败文件下载、模板文件。

追踪到注释代码、旧分支、未被当前链路调用的方法时，不要直接当成业务事实。只有能确认被当前页面入口或当前业务链路调用的代码，才回写业务文档；否则放入源码映射的差异记录或待确认问题。

### 4. 分类证据

每个发现都要分类：

| 分类 | 判定标准 | 写法 |
|------|----------|------|
| 页面已提供功能 | 页面实采、操作手册或当前入口能直接触发 | 写入功能模块正文和按钮/流程说明 |
| 页面未提供功能 | 后端存在接口或方法，但页面实采和操作手册没有入口 | 不写成页面能力；只有该边界会影响用户理解时才在业务页简述，否则只留源码映射 |
| 页面未实采能力 | 代码存在分支，页面可能有入口但当前实采未覆盖，例如需要有数据、失败态或特定下拉值才出现 | 不作为已确认功能；写入源码映射“待实采/待确认”，必要时在业务页标明未实采 |
| 其他业务分支 | 代码存在但不属于当前页面或当前业务域 | 不回写业务页；只在源码映射中记录为排除依据或不记录 |
| 注释/旧逻辑/死代码 | 注释代码、未被当前链路调用、替代实现已存在 | 不作为业务结论；必要时在源码映射中说明“未采用” |
| 与页面冲突 | 源码与页面实采、WIKI、RAW 或用户反馈不一致 | 暂停业务文档回写，按冲突流程请用户确认 |
| 运行证据不足 | 静态代码受部署版本、特性开关、配置、数据或运行时装配影响 | 不宣称当前生效；记录所需日志、环境或测试证据 |
| 支撑性代码 | DTO、枚举、工具、Mapper 等只支撑链路，不是独立能力 | 作为证据引用，不单独提升为业务功能 |

只有“页面已提供功能”或不推翻现有事实的补充规则，才可直接回写业务文档。其他分类应进入源码映射的差异或待确认记录。

影响业务结论的人工决策还要登记到 `WIKI/待确认项/`。源码映射记录技术差异，待确认项负责业务决策，两者不得互相替代。

## 源码映射页结构

源码映射按业务对象落位：

- 功能页面：`WIKI/源码映射/<系统>/<菜单>/<页面>.md`，直接镜像功能模块中的系统/菜单/页面，不增加“功能模块”一层；
- 概念：`WIKI/源码映射/概念/<概念>.md`；
- 操作流程或后台链路：`WIKI/源码映射/操作流程/<流程>.md`。

页面、概念、流程或动作级内容拆成独立 Markdown，避免单文件过大。

映射正文使用 `type: source-map`，并引用目标业务对象已有的具体 RAW 证据。没有业务对象或永久证据支撑时，候选先留 `.work/source-analysis/`，不得用 `type: index` 绕过 source。

建议结构：

```markdown
---
title: 页面名源码映射
type: source-map
source: |
  RAW_SOURCES/截图/示例系统/示例菜单/示例页面/页面证据.png
date: YYYY-MM-DD
updated: YYYY-MM-DD
tags: [源码映射, 示例系统]
related: [页面名]
---

# 页面名源码映射

| 仓库 | 分支 | 完整 commit | 工作区状态 / 指纹 | 作用 |
|------|------|-------------|---------------------|------|

## 页面动作到源码入口

| 页面动作 | 页面接口 | 后端入口 | 关键逻辑 | 证据分类 | 业务结论去向 |
|----------|----------|----------|----------|----------|--------------|

## 字段 / 枚举 / 状态映射

| 页面字段或状态 | 源码对象 | 处理规则 | 证据分类 | 业务结论去向 |
|----------------|----------|----------|----------|--------------|

## 已回写业务结论

- ...

## 页面未提供或代码-only 能力

- ...

## 待确认问题

- ...
```

如不是页面，可把“页面动作”替换为“功能动作”“接口链路”或“规则入口”，但必须保留源码版本、证据分类、业务结论去向和待确认问题。

配置类、规则类或基础数据类页面的映射页建议额外保留“下游使用场景”或“匹配口径”小节：

```markdown
## 下游使用场景

| 使用场景 | 调用入口 | 匹配/取数口径 | 证据分类 | 业务结论去向 |
|----------|----------|---------------|----------|--------------|
```

## 回写业务文档

根据结论类型写入：

| 结论类型 | 写入位置 |
|----------|----------|
| 页面字段、按钮、弹窗、导入、保存、删除、批量操作规则 | `WIKI/功能模块/...` |
| 跨页面概念、校验口径、枚举含义、算法原则、状态机 | `WIKI/概念/...` |
| 操作顺序、排障路径、提交前后流程 | `WIKI/操作流程/...` |
| 接口、类、方法、代码版本、追踪路径 | `WIKI/源码映射/...` |

回写时优先“就近整合”：

- 字段规则补进字段说明。
- 按钮规则补进按钮或弹窗说明。
- 页面执行规则补进“后台处理规则”“执行口径”等业务小节。
- 配置类、规则类或基础数据类页面要区分“页面维护规则”和“下游使用规则”：保存校验写在字段/按钮附近，下游匹配写在“下游使用场景”“后台生成口径”“匹配口径”等小节。
- 同一配置被多个页面或流程消费时，必须按消费场景拆分说明，不能把一个场景的规则写成全局规则。
- 跨页面规则补进概念或流程页，并在页面文档中保留简短入口链接。

不要在页面末尾堆一个泛化的“源码提取补充”章节，把所有代码发现塞进去。代码路径细节留在源码映射页。

### 原子 Finding 与融合动作

大型任务或同一对象发现较多时，把每条结论拆成原子 finding，并记录：

| 字段 | 要求 |
|------|------|
| finding ID | 稳定编号 |
| 业务结论 | 不包含代码路径的可读结论 |
| 目标文件与目标标题 | 结论应进入的既有业务位置 |
| 融合动作 | 使用受控动作 |
| 旧口径处理 | 保留、改写、删除、并列差异或阻断 |
| 源码映射 | 技术证据所在页 |
| 阻塞与复核 | 冲突原因和验证方式 |

受控动作：

- `rewrite-existing`：修正原字段、按钮、状态或流程口径；
- `merge-table`：合并到已有字段、规则或状态表；
- `new-business-section`：确有独立业务主题时新增业务小节；
- `link-only`：业务正文已有正确结论，只补源码映射链接；
- `blocked`：存在冲突，暂停业务回写。

禁止使用 `append-source-section`：不得在页面末尾追加统一的源码专区来逃避语义融合。

## 冲突暂停确认

如果源码与现有知识库、页面实采、RAW 或用户反馈冲突，必须暂停业务文档回写。

冲突包括：

| 冲突类型 | 示例 |
|----------|------|
| 源码与 WIKI 结论相反 | WIKI 写全量覆盖，代码显示按条件更新 |
| 源码与页面实采不同 | 页面提示必填，代码允许为空 |
| 代码存在但页面未暴露 | 后端有接口，页面和手册都没有入口 |
| 源码与用户反馈不同 | 用户确认必填，代码当前未校验 |
| 多仓库口径不同 | 上游保存字段，下游处理忽略或重算 |
| 本地源码疑似非线上版本 | 页面行为与本地代码不一致 |

冲突清单格式：

```markdown
发现以下冲突，先不直接改业务文档：

| 编号 | 主题 | 现有知识库/页面说法 | 源码证据 | 影响范围 | 建议处理 |
|------|------|----------------------|----------|----------|----------|
| C1 | ... | ... | 仓库@commit: 文件/类/方法 | ... | 保留页面口径 / 改为源码口径 / 标为待确认 |
```

处理要求：

1. 先写入源码映射页的“待确认问题”或“冲突记录”。
2. 在 `WIKI/待确认项/` 登记业务决策、验证方式和影响文档。
3. 向用户说明冲突来源、影响范围和建议处理方式。
4. 等用户确认后，再更新功能模块、概念或流程文档。
5. 用户暂不确认时，业务文档保留原说法，源码映射保留冲突记录，对象状态为 `blocked-conflict`。

## 不确定性处理

- 配置类逻辑要区分“配置保存时的校验”和“配置被下游消费时的匹配”；两者可能使用不同字段、不同优先级、不同去重策略。
- 上游保存的字段不等于下游一定消费；下游忽略、重算、转换或兜底的字段应单独说明。
- 本地源码可能不是线上版本，涉及页面行为差异时要标注“基于本地源码版本”，并优先保留页面实采口径。
- 注释代码、旧分支、未被当前入口调用的方法不能直接写入业务页。

## 索引、日志和验证

仅知识库维护模式完成后更新：

- `WIKI/源码映射/` 相关各级 `_index.md`。
- 被回写的 `WIKI/功能模块/`、`WIKI/概念/` 或 `WIKI/操作流程/` 相关 index。
- `WIKI/index.md`。
- `WIKI/log.md`。

日志每次最多两条 bullet，只写维护范围、关键业务结论或遗留边界。仓库、分支、commit 和技术路径写入源码映射，不在 log 复制长清单。

运行：

```bash
python3 skills/lint/scripts/lint_wiki.py
python3 skills/lint/scripts/lint_wiki.py --audit
git diff --check
```

默认 lint 失败时不得返回 `published`。Audit 提示单独报告，不自动阻断发布。

## 调度回执

由 `orchestrate-source-maintenance` 分配任务时，完成后返回：

- 覆盖行 ID 和实际分析边界；
- 仓库、分支、commit 和工作区状态；
- 新增或更新的源码映射页；
- 已回写的功能、概念或流程页；
- 能力分类、冲突、缺少的页面证据和未归属入口；
- 默认 lint 与差异检查结果；
- 最终状态。

状态只使用：

| 状态 | 含义 |
|------|------|
| `published` | 映射、业务回写、index、log、lint 和覆盖矩阵回执全部闭环 |
| `needs-source` | 当前仓库缺少必要源码或无法定位入口 |
| `needs-page-evidence` | 代码存在，但页面能力或形态尚未实采 |
| `needs-runtime-evidence` | 静态代码不足以证明当前部署、配置或数据条件下实际生效 |
| `needs-reproducible-source` | 源码工作树无法冻结或复核，不能绑定可重复证据 |
| `blocked-conflict` | 页面、RAW、正式业务口径与源码冲突，尚未裁决 |

`blocked-conflict` 不是完成状态。大型任务只能在明确批准排除并记录理由后，才从当前完成范围扣除。

## 常见错误

- 把源码映射页写成逐行代码注释。
- 业务规则只留在源码映射页，没有回写到业务文档。
- 把代码存在的接口写成当前页面按钮或功能。
- 不记录 commit，导致后续无法判断结论是否仍适用。
- 源码结论和页面实采冲突时静默覆盖页面事实。
- 把所有结论堆到页面末尾，而不是嵌入字段、按钮、流程或概念。
- 新增映射页后忘记同步 index 和 `WIKI/log.md`。
- 配置页只看保存接口，遗漏谁消费它、如何匹配、如何兜底。
- 混用不同消费场景，把一个流程的取数口径写成全局规则。
- 把未实采分支写成已提供功能。代码存在不等于页面可用。
- 把注释代码、旧分支或未调用方法当成当前业务逻辑。
- 在页面末尾追加泛化源码专区，未把 finding 融合到原字段、按钮、状态、流程、权限或下游章节。
- 大型任务越过覆盖行和批次边界，或用扫描文件数替代业务闭环。
- 只通过逐批 lint 就宣称整体完成，没有独立最终语义审计。

## 检查清单

- [ ] 已读 `AGENTS.md`、全局 index、概念 index、源码映射 index 和目标文件。
- [ ] 已区分只读诊断与知识库维护；没有越权写入。
- [ ] 知识库维护模式已完成初始化，`--strict-init` 通过。
- [ ] 本次范围已收敛到一个页面、动作、接口或规则。
- [ ] 如来自大型任务，已读取覆盖行、固定 commit、批次边界和恢复点。
- [ ] 已记录每个源码仓库的分支、完整 commit、工作区状态和必要指纹。
- [ ] 已按实际技术栈选择入口 → 规则 → 结果链路，没有强套分层后端结构。
- [ ] 如对象是配置类、规则类或基础数据类页面，已追踪保存校验和下游使用点。
- [ ] 每个发现已分类，并区分页面能力、运行证据不足、支撑性代码、其他业务分支、旧逻辑或冲突。
- [ ] 同一配置的多消费场景已拆分说明，没有混用取数口径。
- [ ] 源码映射页已记录证据链，业务结论已就近回写。
- [ ] 每个 finding 已选择受控融合动作，没有使用 `append-source-section`。
- [ ] 冲突已暂停确认，没有静默覆盖页面事实。
- [ ] index、两条以内 log、默认 lint、Audit 复核和 `git diff --check` 已完成。
- [ ] 调度任务已返回覆盖矩阵回执和受控终态。
