# 业务知识库查询流程

每次回答业务问题前，通过分层索引定位相关文档，迭代检索直到信息充分。

## 总体流程

```text
第一步：理解问题，提取关键词
第二步：分层读 Index，确定候选文档集合
第三步：迭代检索，最多 3 轮
第四步：综合回答，注明来源
```

## 第一步：理解问题

提取：

- 主题关键词：系统、角色、概念、页面、动作、状态。
- 问题类型：概念类、流程类、功能模块类、配置类、故障类。
- 输出类型：位置路径、操作步骤、规则说明、字段列表、差异对比。

## 第二步：Index 预热

每次查询前先读：

| 文件 | 作用 |
|------|------|
| `WIKI/index.md` | 全库结构入口 |
| `WIKI/概念/概念_index.md` | 概念关系入口 |

然后按问题类型读取：

| 问题类型 | Index 文件 |
|----------|------------|
| 概念类 | `WIKI/概念/概念_index.md` |
| 功能模块类 | `WIKI/功能模块/功能模块_index.md` |
| 操作流程类 | `WIKI/操作流程/操作流程_index.md` |
| 源码/隐藏规则类 | `WIKI/源码映射/源码映射_index.md` |
| 不确定 | 先回到 `WIKI/index.md` |

## 第三步：迭代检索

- 最多 3 轮。
- 每轮选择 2-5 个候选文档。
- 优先读取最相关局部，不要无目的全量读取。
- 信息不足时扩展同义词、旧称、页面名、按钮名。
- 如 WIKI 不足以回答界面细节，可查看 frontmatter `source` 指向的 RAW。
- 如 WIKI 不足以回答隐藏校验、接口链路或代码原因，可查看 `WIKI/源码映射/`，但要说明结论基于源码版本。

## 第四步：回答规范

- 先给结论，再给依据。
- 明确引用 WIKI 路径。
- 如引用 RAW，注明 RAW 路径。
- 如果知识库没有完全匹配信息，说明信息边界，不编造。
