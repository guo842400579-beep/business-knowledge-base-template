# 业务知识库查询流程

每次回答业务问题前，先通过分层索引定位文档，再按证据层级迭代检索。回答必须说明来源和边界：WIKI 能回答什么，RAW 或源码映射补充了什么，哪些仍需确认。

## 核心边界

- 本 skill 只回答当前知识库业务域内的问题。
- 不要把其他知识库、历史项目或个人记忆里的业务结论迁入当前知识库。
- 如果用户问题明显超出当前业务域，先说明边界，再给出可继续检索或维护的路径。
- 如果查询过程中发现 WIKI 缺漏，不要直接编造；应提示可用 `record-feature-to-wiki`、`extract-source-to-wiki` 或 RAW 提取流程补齐。

## 必读入口

查询前必须先读：

| 文件 | 作用 |
|------|------|
| `AGENTS.md` | 确认业务边界、目录规范、证据优先级和维护铁律 |
| `WIKI/index.md` | 建立全库结构视图 |
| `WIKI/概念/概念_index.md` | 建立术语、概念和同义词上下文 |

如果用户使用口语、截图化描述或旧称，先查 `WIKI/概念/术语对照表.md`；如果该文件不存在，则在 `WIKI/index.md`、模块 index 和文件名中查找可能对应项。

## 总体流程

```text
理解问题 -> 读取全局入口 -> 定位候选文档 -> 最多 3 轮迭代检索 -> 综合回答并注明来源
```

## 1. 理解问题

提取：

- 主题关键词：系统名、角色、概念、页面、Tab、按钮、字段、状态、接口或规则名。
- 同义词和旧称：用户口语、页面可见名称、文档正式名称、源码字段名。
- 问题类型：概念类、功能模块类、操作流程类、配置规则类、源码/隐藏逻辑类、故障排查类、历史记录类。
- 期望输出：位置路径、操作步骤、规则说明、字段列表、按钮行为、差异对比、证据链、待确认问题。

## 2. 分层读 Index

根据问题类型选择 index：

| 问题类型 | 优先读取 |
|----------|----------|
| 概念、术语、规则口径 | `WIKI/概念/概念_index.md` |
| 页面、菜单、按钮、字段、Tab | `WIKI/功能模块/功能模块_index.md` |
| 跨页面操作、角色协作、排障路径 | `WIKI/操作流程/操作流程_index.md` |
| 接口、隐藏校验、代码链路、算法逻辑 | `WIKI/源码映射/源码映射_index.md` |
| 不确定或跨模块 | 回到 `WIKI/index.md`，再深入子目录 index |

Index 只用于定位，不要把 index 摘要当成完整事实。定位后读取最相关的 2-5 个具体文档。

## 3. 迭代检索

最多 3 轮。每轮执行：

1. 用关键词定位候选文档或局部。
2. 读取最相关文件的相关章节，不做无目标全库通读。
3. 判断信息是否足够回答用户问题。
4. 如不足，扩展同义词、页面名、按钮名、字段名、接口名或源码对象名后继续。
5. 达到 3 轮仍不足时，停止扩展并说明当前知识库缺口。

优先使用 `rg` 定位关键词；读取文件前先确认路径属于当前知识库。

## 4. 何时查看 RAW

以下情况可查看 frontmatter `source` 指向的 `RAW_SOURCES/`：

- 用户追问按钮位置、弹窗、下拉、空状态、宽表列头等界面细节。
- WIKI 描述不够明确，需要回看截图、操作手册、模板或专家原话。
- 用户质疑现有结论，需要追溯原始证据。
- 文档提到“示例数据”，需要确认是否只是样例而非全集。

查看 RAW 后回答要注明 RAW 路径。RAW 是原始证据，只读不改；若发现 RAW 与 WIKI 冲突，按“待确认”处理，不直接覆盖结论。

## 5. 何时查看源码映射

以下问题优先查看 `WIKI/源码映射/`：

- “为什么系统这样判断”
- “提交后调用了什么接口”
- “隐藏校验、状态机、枚举或算法规则是什么”
- “页面没有展示但后台是否有处理”

回答源码相关问题时必须说明：

- 结论基于哪个源码映射页。
- 映射页记录的仓库、commit 或版本是否明确。
- 该能力是页面可见、页面未观察到，还是仅代码存在。

不要把代码存在的接口直接说成用户当前页面可用功能。

## 6. 综合回答规范

回答结构：

```markdown
结论：...

依据：
- WIKI/功能模块/...
- WIKI/概念/...
- RAW_SOURCES/...（如查看）
- WIKI/源码映射/...（如查看）

边界：
- 当前知识库未覆盖...
- 该结论基于本地源码版本...
- 该字段/数据只是示例...
```

要求：

- 先给结论，再给依据。
- 路径写清楚，便于用户复查。
- 明确信息边界，不编造未记录内容。
- 页面事实优先于源码推断；源码冲突只能标为待确认。
- 示例数据不能回答成枚举全集。

## 7. 查询转维护

查询中发现缺口时，按缺口类型建议后续动作：

| 缺口 | 后续 skill |
|------|------------|
| 页面字段、按钮、弹窗、截图缺失 | `record-feature-to-wiki` |
| 隐藏校验、接口链路、算法规则缺失 | `extract-source-to-wiki` |
| RAW 已存在但未结构化 | 按 `AGENTS.md` 的 RAW 提取模式维护 |
| 断链、frontmatter、index 或表格问题 | `lint` |

如果用户明确要求“顺手补上”，先切换到对应维护流程，并完成 WIKI/index/log/lint 闭环。

## 查询检查清单

- [ ] 已读 `AGENTS.md`、`WIKI/index.md`、`WIKI/概念/概念_index.md`。
- [ ] 已把用户口语映射到正式概念、页面或路径。
- [ ] 已读取最相关的 2-5 个文档，而不是只看 index。
- [ ] 信息不足时已最多迭代 3 轮。
- [ ] 如查看 RAW 或源码映射，已在回答中注明来源。
- [ ] 已区分页面可见事实、RAW 补充、源码推断和待确认问题。
